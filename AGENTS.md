# AGENTS.md

Development standards for the opencode-go-sdk project. Read this before making changes.

## Project overview

This is a Go client SDK for the OpenCode server's HTTP API. It has two layers:

1. **Generated code** (`opencode.gen.go`) — produced by [oapi-codegen](https://github.com/oapi-codegen/oapi-codegen) from the committed `opencode-spec.json` (a snapshot of the upstream OpenAPI spec).
2. **Hand-written code** (`auth.go`, `prompt.go`, `sse/`) — convenience helpers for auth, prompt building, and SSE event streaming.

The upstream spec is OpenAPI 3.1.0; `scripts/pull-and-normalize.go` down-converts 3.1 constructs (numeric `exclusiveMinimum`, `anyOf` with `type: null`, dotted schema names) to 3.0-compatible JSON before generation.

## The separation rule (critical)

**Generated files must never be hand-edited. Custom code must only depend on generated exported types.**

- **Generated files** end in `.gen.go` and carry the `// Code generated by oapi-codegen DO NOT EDIT.` header.
- **Custom files** are plain `.go` files (`auth.go`, `prompt.go`, `sse/*.go`).
- Custom code may reference **exported** types, methods, and functions from the generated package (`opencode.Event`, `opencode.Client`, `opencode.SessionPromptJSONBody`, etc.). It must never reference unexported fields or `internal/` packages from generated code.
- The regen CI guardrail asserts that `make generate` only touches `*.gen.go` and `opencode-spec.json`. If it touches custom files, CI fails.
- A test (`separation_test.go`) asserts that no custom `.go` file contains the "DO NOT EDIT" marker.

### Defensive access to volatile types

The `Event` and `GlobalEvent` unions grow over time as upstream adds event variants. Custom SSE helpers handle this without breaking:

- `EventType(e)` returns the variant's `type` string; for unknown/missing variants it returns `""` (never panics).
- `EventAs[T](e)` returns a typed `*T` for known variants and `ErrUnknownEventType` for unknown ones.

**Adding a variant upstream never breaks existing custom code.** It produces a new `type` string that caller code can opt into later.

### Compile-time tripwire

If upstream *renames or removes* a generated field that custom code references, the build fails loudly. This is a feature — the regen PR's CI fails, the PR is not mergeable, and a human fixes the custom code in the same PR. There is no attempt to silently absorb breaking renames.

## Build and test commands

```bash
make pull       # fetch upstream spec from anomalyco/opencode dev branch
make generate   # normalize spec + run oapi-codegen → opencode.gen.go
make test       # go build ./... && go test ./...
make regen      # pull + generate + test (the full refresh loop)
make check      # verify generated files are in sync with the spec
```

Always run `make test` before committing. If `make generate` changes `opencode.gen.go`, commit the generated file alongside any custom changes.

## Code style

- `gofmt -s` and `go vet` are required.
- Use `errors.Is`/`errors.As` for error handling, not string comparison.
- Context-first parameters (`ctx context.Context` as the first arg).
- Godoc comments on all exported symbols.
- Generated files (`*.gen.go`) are excluded from linting.
- No external dependencies beyond `github.com/oapi-codegen/runtime` (required by generated code).

## Conventional Commits (required)

This project uses [release-please](https://github.com/googleapis/release-please) for automated versioning. It reads Conventional Commits to determine the version bump:

| Commit type | Version bump |
|---|---|
| `feat:` | minor |
| `fix:` | patch |
| `BREAKING CHANGE:` or `feat!:` | major |
| `refactor:`, `docs:`, `chore:`, `test:`, `build:`, `ci:` | none (but listed in changelog) |

**Regens always use**: `feat: regen from upstream opencode spec`

## Adding endpoints

Never hand-edit generated files. Instead:

1. Run `make pull` to fetch the latest upstream spec (or `make pull-local` against a running server).
2. Run `make generate` to regenerate `opencode.gen.go`.
3. If you need a convenience helper for a new endpoint, add it in a plain `.go` file (e.g. `prompt.go`) that references the generated types.
4. Commit `opencode-spec.json`, `opencode.gen.go`, and your custom helper.

## Releasing

Never tag manually. release-please handles it:

1. Conventional Commit PRs merge to `main`.
2. release-please automatically maintains a "release PR" (`chore(main): release X.Y.Z`) with accumulated changelog and version bump.
3. Merge the release PR → release-please tags `vX.Y.Z`, updates `.release-please-manifest.json`, and creates a GitHub Release.
4. `go get github.com/codeownersnet/opencode-go-sdk@vX.Y.Z` resolves to the tag.

## Updating the spec snapshot

```bash
make pull       # fetch latest upstream spec → opencode-spec.json
make generate   # regenerate opencode.gen.go
make test       # verify
```

Commit both `opencode-spec.json` and `opencode.gen.go`. The spec diff shows what changed upstream; the generated code diff shows the impact on the SDK.
